一、查看改动

1、查看当前仓库状态

git status

有需要提交的改动 / 冲突时，会有提示。

2、查看改动的内容

git diff 文件名

二、版本回退

1、查看log

git log

// 获取简要的 log
git log --pretty=oneline

2、版本回退

1）向前回退

// 回退到上一个版本
git reset --hard HEAD^

// 回退到上上的版本，以此推类
git reset --hard HEAD^^

// 回退到前100个版本，以此推类
git reset --hard HEAD~100

2）向后回退

// 获取版本号
git reflog

// 回退到对应版本
git reset --hard 版本号

三、撤销 / 删除

1、撤销修改（commit之前）

1）手动将文件修改回去；

2）按上面的方法，回退版本；

3）使用撤销命令。

git checkout --文件名

checkout –文件名的意思是把文件在工作区做的修改全部撤销，有两种情况。

  • 文件修改，还未放到暂存区，撤销即回到和版本库一样的状态；

  • 已放入暂存区，后又做了修改，撤销即回到添加暂存区后的状态。

注意：“– 文件名”，是2横，且与文件名间有空隙。

2、删除文件

1）手动删除；

2）终端命令；

rm test3.txt

若想在版本库中删除，删完commit即可。

四、解决冲突

1、查看冲突的文件

git status

打开冲突文件，修改 «««< ======= »»> 中的冲突部分，再提交

2、查看分支合并图

git log --graph

3、bug 分支（stash）

当你接到一个修复一个代号101的bug的任务时，很自然地，你想创建一个分支issue-101来修复它，但是，等等，当前正在dev上进行的工作还没有提交.

并不是你不想提交，而是工作只进行到一半，还没法提交，预计完成还需1天时间。但是，必须在两个小时内修复该bug，怎么办？

幸好，Git还提供了一个stash功能，可以把当前工作现场“储藏”起来，等以后恢复现场后继续工作：

git stash

现在，用git status查看工作区，就是干净的（除非有没有被Git管理的文件），因此可以放心地创建分支来修复bug。

bug 修改完成后，要继续之前打断的工作。

查看被保存的现场：

git stash list
// stash@{0}: WIP on dev: 6224937 add merge

恢复现场：

git stash apply   // 恢复
git stash drop    // 删除记录

相当于：

git stash list

当有多个 stash 时，可以恢复到指定的：

git stash apply stash@{0}
