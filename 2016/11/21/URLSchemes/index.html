是什么

weixin://dl/moments

说明：

类比URL，weixin:// 指定了是哪个 App，而 dl/moments 指定了具体的功能，比如此处是微信朋友圈的意思。

注： URL Schemes 没有唯一性，也就是说，多个App可以由同样的 URL Schemes；

iOS 上的 URL Schemes 因为没有统一标准，所以非常难猜，通过猜来获取 iOS 应用的 URL Schemes 是不现实的。

使用形式

1、基本

YSDevScheme://

如何获取一个应用的基本 URL Schemes

2、复杂

打开某个功能（带参数）

// 打开某个功能界面
fantastical2://parse?sentence


// 打开某个功能，并且带参数过去
fantastical2://parse?sentence=事件描述

加 jumpback，可返回

eudic://dict/想查的单词?jumpback=指定URL

如 B App 的 URL Schemes 如上，在 A App 中拉起 B，然后点击返回，回到指定的 URL（可以是 A，也可以是其他的 App）。

3、变形

利用 URL Schemes 的规则和 iOS 系统的一些内置功能，来扩充复杂 URL Schemes，使其中需要输入字符或参数的部分可以预设或输入后再跳转。

1）可输入的参数

// 在 Launch Center Pro 里，输入框表示为[prompt]，你只要在变量部分填入这个输入框的表述方式，就会在运行动作时出现输入框让你输入内容，所以我们可以把上面那个 URL 改为：

OmniFocus:///add?name=[prompt]&note=[prompt]

// 在 Launch Center Pro 里最后一条复制了的文本内容表示为[clipboard]，当你想用欧陆词典直接搜索你刚才复制好了的不知道意思的单词的时候，就可以在 Launch Center Pro 里添加一个这样的动作：

eudic://dict/[clipboard]

2）list 列表

//

onepassword://search/[list|Google=Google|Dropbox=Dropbox|Weibo=Weibo|输入=[prompt]]

// 首先有一个中括号[]，然后括号的最左边是 list 和一个分隔符 |，这三样元素声明了这是一个列表。然后是 Google=Google、微博=Weibo 等常用服务，等号左边，是显示在列表上的名称，而右边是填入 URL 的字符。最后一个 输入=[prompt]，左边同样是列表上的名称，右边是前面提到的输入框。最后这一部分是列表中没有你想要搜索的项目的时候手动输入用的。

延伸

1、x-callback-URL

如果我们还想让应用根据不同的结果有对应的反应，就要用到 x-callback-URL。比如当上一个 URL Schemes 运行成功以后，我是要回到跳转前的应用？还是要接另一个动作（接上另一段 URL Schemes，打开另一个应用的某个功能）？无论是跳转回上个应用还是打开另一个动作，只要你在运行完一个 URL Schemes 后还想再利用一段新的 URL Schemes 做下一件事，就要靠 x-callback-URL

它的固定语法是：

  • 在一个 URL Schemes 后面接&x-success，表示前一个 URL 成功以后下一步做什么；
  • 在一个 URL Schemes 后面接&x-error，表示前一个 URL 失败以后下一步做什么；
  • 在一个 URL Schemes 后面接&x-cancel，表示取消前一个 URL 的操作结果后下一步做什么；
  • 还有一个 &x-source ，比上面3个特殊些。

x-source

在 x-callback-URL 的四大固定语法中，x-source 是个很小众的用法。它一般和 x-success 连接，构成 x-source=选项名&x-success=的格式。

显示在 App 中则是在上一个动作完成后（一般是跳转后）给出一个选项，选项内容是你写在 URL 中的选项名和 取消，选择选项名后触发 x-success 后的 URL 里的动作，取消则停留在当前 App。也就是说，它给了你一个选择，在这一步到底是继续下一个动作还是停留在当前 App。

实例：用 Drafts 给Fantastical 一次性添加多个事件

前提：

fantastical 可以识别自然语言，从一句话里提取出事件、时间自动添加到默认日历中。

draft 可将多个 URL 绑在一起发送给 fantastical，实现一次添加多个事件。

// 把多个发送事件到 Fantastical 的 URL 绑在了一起，做到一个运行成功以后运行下一个，直到运行完，实现这样的 URL 就只有 x-callback-URL 才可以，具体的 URL 可以说很复杂，是这样的：

fantastical2://x-callback-url/parse?sentence=[[line|1]]&add=1&x-success=&x-cancel=

分析：

  • fantastical2://x-callback-url/parse?sentence= 比 fantastical2://parse?sentence= 多了一截儿 x-callback-url/，这就是在声明，各单位注意，我下面的 URL 要用到 x-callback-URL 了。
  • 紧接着是 [[line|1]]，这一段是 Drafts 的变形 URL，意味取文本中第一行的值。
  • add=1是 fantastical 的语法，再加一项任务。
  • &x-success=是 x-callback-URL 的标准声明，即成功后做什么。
  • x-callback-URL中出现在固定语法之后的``````意味着，双括号后是一个完整的动作，这个动作在 success/error/cancel 后执行。
  • runAction?是 Drafts 的运行某个动作的起始声明，代表 Drafts 要运行一个动作了，后面的不远处肯定能找到要运行的动作的名称。
  • text=[[line|2..]]并不是动作名称，而是动作的内容。text= 是 Drafts 里的一个语法，等号后面决定了该动作使用的文本是什么。然后[[line|2...]]这句跟我们之前见过的[[line|1]]不太一样，如果数字 2 后面没有点点点，就特指的是第一句，但是如果有点点点了，意思就是对下面每行都重复该动作。
  • &allowEmpty=NO这一句在 Drafts 的语法中意味：不允许发送空内容。
  • &action=Events%20in%20Fantastical-Quick提出了动作名，动作名是Events%20in%20Fantastical-Quick。
  • %20涉及到什么时候要在 URL 中使用编码，什么时候不使用的问题。
  • &x-cancel=取消任务以后会回到（实际是打开）Drafts，但什么都不做。

这一整段的 x-callback-URL，如果想正常工作，必须要保存为 Drafts 中的一个动作，动作名要和 URL 中提到的 Events in Fantastical-Quick 一致，改的话两个都要改。这是最复杂的 URL Schemes 的一种形式之一：在一个 App （Fantastical）的 URL 中嵌套另一个 App（Drafts） 的 URL，同时 Drafts 的 URL 本身指涉的动作实际上还是不断自指的。

URL编码（Encode）

什么时候要编码？

URL 中的字符可以分为两类，一部分可以正常显示，一部分不能正常显示。在链接中正常显示，比如字母、数字还有/等一部分符号。除此之外，全部不能正常显示，需要进行编码才能够作为 URL 的一部分出现。比如空格，在 URL 中就必须表示为 %20转换的规则不用深究，网上有很多工具（比如 URL 解码工具）提供 URL 的编码和解码功能，可以把编码过的乱七八糟的 URL 解码为我们看得清爽的字符：

为什么有的 App 不需要编码？

因为那些不用编码的时候，是 App 私下替你编了。
